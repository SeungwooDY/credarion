# Changelog — 2026-04-20

## Phase 3: Invoice Processing (Week 7)

### Backend — Invoice Processing Module

**New models** (`backend/app/models.py`):
- `Invoice` — OCR-extracted fapiao records with per-field confidence scoring, status workflow, supplier matching
- `InvoiceLineItem` — per-line details with PO/material number for future reconciliation linking

**Migration** (`0004_invoice_processing.py`):
- Creates `invoices` and `invoice_line_items` tables with indexes on org_id, status, invoice_number, po_number, material_number

**New module** (`backend/app/invoicing/`):
- `file_storage.py` — Save/retrieve uploaded files to local disk (MVP; swap to S3 later)
- `ocr_extractor.py` — Claude Vision API extraction for 增值税专用发票 and 增值税普通发票. Extracts supplier_name, invoice_number, date, amounts, VAT, currency, line items. Per-field confidence scoring (0.0–1.0), overall confidence = min of all fields, flags needs_review if any field < 0.80
- `supplier_matcher.py` — Match extracted supplier name to DB (exact match, then contains match)
- `schemas.py` — Pydantic models for all request/response types, status transition validation

**New router** (`backend/app/routers/invoices.py`):
- `POST /upload` — Batch upload (multiple files), creates Invoice records with status=received
- `POST /{id}/extract` — Trigger OCR, populate fields + line items, auto-match supplier, set status=extracted
- `GET /` — List with filters (status, supplier_id, needs_review, date range, pagination)
- `GET /{id}` — Detail with nested line items
- `PUT /{id}` — Update fields (manual correction), re-attempts supplier matching
- `PUT /{id}/status` — Transition status with validation (received→extracted→matched→approved→paid)
- `GET /{id}/line-items` — Get line items
- `PUT /{id}/line-items/{line_id}` — Update specific line item

**Config** (`backend/app/config.py`):
- Added `invoice_upload_dir` (default: "uploads/invoices")
- Added `invoice_ocr_model` (default: "claude-haiku-4-5-20251001")

**Tests** (40 new, 150 total passing, 1 known failure):
- `test_invoice_ocr.py` (12 tests): extraction success, low confidence, PDF/image content blocks, malformed JSON, markdown code blocks, API failure, unsupported types, file not found, empty response, file storage
- `test_invoice_api.py` (28 tests): upload single/multiple/rejected types, extract success/not found/wrong status/no API key, list/filter/pagination, detail, field updates with supplier re-matching, status transitions valid + invalid, line items CRUD, supplier matcher exact/contains/reverse/no match/empty

## Frontend — Next.js Skeleton UI

Replaced boilerplate Create Next App with a connected multi-page application.

**Infrastructure:**
- `next.config.ts` — API proxy rewrite (`/api/*` → `http://localhost:8000/api/*`)
- `app/lib/api.ts` — Lightweight fetch helper (apiFetch, apiPost, apiPut, apiUpload)

**Shared components:**
- `Sidebar` — Navigation with active state highlighting (Dashboard, Ingestion, Reconciliation, Invoices, Settings)
- `PageHeader` — Title + description header
- `StatusBadge` — Color-coded status pills for all status types

**Pages:**
- `/` (Dashboard) — API health check, org count, quick-action cards linking to all sections, org table
- `/ingestion` — Dual-panel: GRN upload with SSE progress streaming + supplier statement upload with supplier/period selectors
- `/reconciliation` — Org/supplier/period selectors, run button, 6-stat summary grid, full results table with match type, status badges, deltas, confidence
- `/invoices` — Upload section (multi-file), invoice list with status/review filters, inline extract button, detail side panel with confidence scores + line items + status transitions
- `/invoices/[id]` — Full detail page: extracted fields with per-field confidence %, edit mode, line items table, action buttons, metadata sidebar
- `/settings` — Org management (list + create) and reconciliation config editor (tolerances, AI toggle, token limits)

**Design:**
- Tailwind CSS v4, Geist font family
- Light theme with CSS custom properties (--background, --foreground, --muted, --border, --accent)
- Minimal, functional — skeleton for Richard to build on
